# SmartSync 开发文档

## 1. 项目简介

SmartSync 是一个基于 Spring Boot 的后端学习项目，场景是“医院分布式终端通过 RFID 手环读取患者档案”。

这个项目目前主要解决三件事：

1. 终端注册、审核、登录，并通过 JWT 维持认证状态。
2. 患者档案的增删改查，以及 RFID 手环和患者的绑定关系。
3. 终端刷 RFID 后，按手环标识快速拉取对应患者档案。

从代码量和模块边界来看，它很适合拿来练习这些基础能力：

- Spring Boot 项目结构
- RESTful API 设计
- Spring Security + JWT
- MyBatis-Plus 基础用法
- Flyway 数据库迁移
- 统一响应和统一异常处理
- DTO / Entity / VO 分层

## 4. 核心业务模型

### 4.1 终端 Terminal

终端是医院里的读卡设备或工作站，核心字段有：

- `terminalCode`：终端唯一编码，也是登录账号
- `secretHash`：终端密钥的 BCrypt 哈希
- `status`：终端状态，`0` 待审核、`1` 启用、`2` 禁用

业务规则：

- 终端先注册，再由管理端审核启用
- 只有 `status = 1` 的终端可以登录
- 登录成功后会更新 `lastLoginAt`

### 4.2 患者 Patient

患者表保存的是“患者这个人”的长期档案，不保存挂号、住院、诊断等就诊过程数据。

核心字段有：

- `idCardNo`：身份证号，业务唯一
- `rfidUuid`：当前绑定手环的 12 位 UUID，可为空
- `medicalHistory`：长期病史
- `emergencyContactName` / `emergencyContactPhone`

业务规则：

- 一个身份证号只能对应一份有效档案
- 一个 RFID 只能绑定一个患者
- RFID 为空表示未绑定手环或手环已回收

### 4.3 RFID

项目里的 RFID 不是直接存 13 位完整码，而是分成两部分理解：

- 前 12 位：UUID
- 第 13 位：服务端根据 HMAC 计算出来的校验位

这样做的目的是防止伪造。终端上传 13 位 RFID 后，服务端会先验签，再查数据库。

## 5. 启动流程

### 5.1 配置文件

主配置在：

- `src/main/resources/application.yaml`
- `src/main/resources/application-dev.yaml`

关键配置项：

```yaml
spring:
  profiles:
    active: dev

smartsync:
  jwt:
    secret: ${SMARTSYNC_JWT_SECRET:...}
    expire-hours: 2
  rfid:
    hmac-key: ${SMARTSYNC_RFID_HMAC_KEY:...}
```

你需要特别关注这 3 类配置：

1. MySQL 连接信息
2. JWT 密钥
3. RFID HMAC 密钥

### 5.2 本地启动步骤

1. 创建数据库 `smartsync`
2. 修改 `application-dev.yaml` 里的数据库账号密码
3. 按需设置环境变量：

```bash
export SMARTSYNC_JWT_SECRET="your-jwt-secret-at-least-32-bytes"
export SMARTSYNC_RFID_HMAC_KEY="your-rfid-hmac-key-at-least-16-bytes"
```

4. 启动项目：

```bash
./mvnw spring-boot:run
```

启动后可以访问：

- Swagger UI: `http://localhost:8080/swagger-ui.html`
- OpenAPI JSON: `http://localhost:8080/v3/api-docs`

### 5.3 数据库初始化

Flyway 会在启动时自动执行 `src/main/resources/db/migration/V1__init_schema.sql`。

当前初始化了两张核心表：

- `terminal`
- `patient`

## 6. 请求主链路

### 6.1 终端注册与登录

流程如下：

1. 终端调用 `POST /api/auth/register`
2. 服务端校验 `terminalCode` 是否重复
3. 服务端将 `secretKey` 用 BCrypt 加密后保存
4. 新终端默认状态为 `0`，需要审核
5. 管理端调用 `PUT /api/terminal/{id}/status` 改为 `1`
6. 终端调用 `POST /api/auth/login`
7. 服务端校验密钥并签发 JWT

### 6.2 受保护接口访问

除了 `/api/auth/**` 和 Swagger 文档，其它接口都需要 JWT。

请求头格式：

```http
Authorization: Bearer <token>
```

请求进入系统后的认证流程：

1. `SecurityConfig` 配置安全规则
2. `JwtAuthenticationFilter` 从请求头提取 JWT
3. `JwtUtil` 负责验签和解析 claims
4. 解析成功后把 `TerminalPrincipal` 放入 `SecurityContext`
5. 后续接口即可视为“当前终端已登录”

### 6.3 按 RFID 拉取患者档案

这是项目最核心的一条业务链路：

1. 终端调用 `POST /api/patient/fetch-by-rfid`
2. 请求先通过 JWT 认证
3. `RfidUtil.verify()` 校验 13 位 RFID 是否合法
4. 验证通过后拿到前 12 位 UUID
5. `PatientServiceImpl` 按 `rfidUuid` 查询患者
6. 找到后组装 `PatientVO` 返回

## 7. 分层设计

项目基本遵循经典分层：

- `controller`：接收请求，做参数绑定，返回统一结果
- `service`：业务逻辑
- `mapper`：数据库访问
- `entity`：数据库实体
- `dto`：请求参数和响应对象

建议你重点理解下面几组对象的区别：

- `Entity`：和数据库表结构强相关
- `DTO`：接口入参
- `VO`：接口出参
- `Result`：最外层统一响应壳

一个典型请求通常是：

`Controller -> Service -> Mapper -> DB -> Service -> Controller -> Result`

## 8. 统一返回与异常处理

### 8.1 统一返回

所有接口都返回：

```json
{
  "code": 200,
  "message": "OK",
  "data": {}
}
```

对应实现：

- `Result`
- `ResultCode`
- `PageResult`

这样做的好处是前端或终端可以统一处理响应结构。

### 8.2 业务异常

业务层通过 `BusinessException` 主动抛出可预期错误，例如：

- 终端编码已存在
- 终端未启用
- RFID 未绑定患者

全局异常处理器 `GlobalExceptionHandler` 负责把异常转换成统一 JSON。

## 9. 数据库设计重点

### 9.1 terminal 表

重点字段：

- `terminal_code` 唯一
- `secret_hash` 存加密后的密钥
- `status` 控制是否允许登录
- `deleted` 用于逻辑删除

### 9.2 patient 表

重点字段：

- `id_card_no` + `deleted` 唯一
- `rfid_uuid` + `deleted` 唯一
- `deleted` 用于逻辑删除

这里的唯一索引配合业务层主动校验，形成“双保险”：

- 业务层先给出友好的错误提示
- 数据库层兜底保证数据一致性

## 12. 当前实现的边界与改进方向

这个项目现在更偏“学习型后端项目”，离生产级还有一些明显差距：

- 终端管理接口暂时没有真正的管理员角色控制
- 敏感字段还没有做加密和脱敏
- 测试覆盖面偏小，主要集中在 `RfidUtil`
- 没有区分“患者档案”和“本次就诊信息”
- 还没有审计日志、操作人记录、权限细分

如果你后面想继续扩展，比较自然的方向有：

1. 增加管理员账号体系和 RBAC 权限模型
2. 给患者信息做字段加密和脱敏
3. 引入住院记录、床位、科室等就诊维度
4. 增加接口测试和集成测试
5. 给终端接口增加防重放、防刷接口能力

## 13. 一句话理解整个项目

你可以把 SmartSync 理解成：

“一个给医院终端设备使用的后端服务，负责让合法终端通过 JWT 登录，再通过带校验位的 RFID 手环，安全地读取患者档案。”
